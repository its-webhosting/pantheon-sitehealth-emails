Managing Cloudflare Caching

For an overview of how Cloudflare caches content, refer to Default Cache Behavior in the Cloudflare documentation. At other institutions, Cloudflare only caches static assets by default; it does not cache HTML (web pages).

The University of Michigan has configured most Cloudflare domains so that they do cache HTML (web pages) unless one or more of the following conditions are met:

  • A recognized login cookie is set:
    • For Turnkey Cloudflare and Self Service Cloudflare, any cookie name beginning with any of the following: SSESS, SESS, phpsessid=, wp-, or wordpress.
    • For Basic Cloudflare, any cookie containing any of the above in either its name or value.
  • The website sends an HTTP response header telling cloudflare not to cache the page.
  • The HTTP status code for the page is something other than 200.
  • The owner of a Self Service Cloudflare or Basic Cloudflare domain disables or deletes the cache rule “UM default: Cache pages for non-logged in users”.

In addition, the university has configured most Cloudflare domains to always respect the duration (TTL) send by the website for how long everything should be cached. This applies not only to HTML (web pages) but also to all static assets, and is different from what is standard for Cloudflare outside the university.

The CF-Cache-Status HTTP response header shows whether Cloudflare is caching a particular resource (page or asset), and if so, how. Refer to Cloudflare Cache Responses for details.

Your website should send appropriate HTTP response headers (including Cache-Control and Expires) to control whether/how Cloudflare caches content. Most websites using a CMS will have a caching plugin or module that sends the correct headers for you (see: Installing the umich-cloudflare plugin for WordPress or Installing the Cloudflare Module for Drupal 9 or 10). For an overview of the caching headers, refer to Origin Cache-Control in the Cloudflare documentation. That page also includes examples of Cache-Control headers to use in various situations.

To view the cache headers that Cloudflare sends to website visitors, click on a request in the Network tab of your web browser’s Developer Tools. Alternatively, you can run the following command to view the headers:

curl -s -v -o /dev/null https://your-website.your-unit.umich.edu/some-page/

If you want to view the headers your website is sending to Cloudflare (before Cloudflare modifies them) then run the following command from a university network or when logged in to the U-M VPN, replacing WEBSERVER_FQDN below with your web server’s DNS name:

curl -s -v -o /dev/null \
--connect-to your-website.your-unit.umich.edu:443:WEBSERVER_FQDN:443 \
https://your-website.your-unit.umich.edu/some-page/

For example, to see what caching headers the web server at chaos.dsc.umich.edu is sending to Cloudflare for the web page https://safecomputing.umich.edu/protect-yourself/be-safe-online?nav run the command:

curl -s -v -o /dev/null \
--connect-to safecomputing.umich.edu:443:chaos.dsc.umich.edu:443 \
https://safecomputing.umich.edu/protect-yourself/be-safe-online?nav

Inspecting the HTTP headers received from Cloudflare by visitors’ web browsers and comparing them to the headers sent to Cloudflare by the web server will allow you to determine whether the cause of a problem is at Cloudflare or at the website behind Cloudflare.
