Installing the Cloudflare Module for Drupal 10 or 11

A Cloudflare plugin for Drupal is necessary to allow Drupal to:

  • Tell Cloudflare what pages and assets to cache and what ones not to cache. This is done using HTTP caching headers.
  • Tell Cloudflare how long to cache different pages and assets. This is done using HTTP caching headers.
  • Tell Cloudflare when Drupal content has been modified so Cloudflare can serve the new version to visitors instead of continuing to serve the old version.
  • Allow the Drupal administrator to tell Cloudflare to purge all its content in its cache for the website so that visitors will get new (updated) content rather than old (cached) content.
Note for Drupal 7 website owners: This document applies only to Drupal 10 and 11. Cloudflare ended support for Drupal 7 in 2016. For Drupal 7 websites, please see the ITS documentation, Configure Caching for Drupal 7 behind Cloudflare for recommendations on how to configure Drupal 7 behind Cloudflare.
Note: Most of the procedural steps required in this document can be completed via the command line. Those that must be done using the Drupal administration user interface (UI) will be noted as such.
Use of Composer and Drush: For sites hosted on U-M Pantheon, these will require the use of the Terminus command-line tool. For other hosting environments, ensure you have Composer and Drush installed according to your hosting service's requirements.

Create a domain-specific Cloudflare API token

Refer to the documentation, Create a Cloudflare API Token to use with a Cloudflare Content Management System module/plugin, for the procedure for creating a Cloudflare API token.

Cloudflare API tokens are usable for all websites in the Cloudflare domain for which the token was created. If you already have a Cloudflare API token for the domain your website is in, you can use that token for all websites in the domain.

Install prerequisites

The Chaos Tools (ctools) and Purge (purge) modules are required before the “Cloudflare” (cloudflare) module can be enabled. Install those first via drush from the command line.

composer require 'drupal/ctools:^4.1'

composer require 'drupal/purge:^3.6'

Install “cloudflare” module

Note: As of this writing, the Drupal 10/11 Cloudflare module is not at GA release status. The Drupal 10/11 module is at “beta” status. For more information, see the Drupal Cloudflare project page.

Install the module via Composer. 

 

Drupal 10/11:

composer require 'drupal/cloudflare:^2.0@beta' -W

Install "HTTP Cache Control" module

This module adds more configuration options to the Configuration --> Development --> Performance page.  In particular, this allows configuration of the "must-revalidate" and "s-maxage" Cache-Control headers.  Install the module via Composer.  Module project page is here.

composer require 'drupal/http_cache_control:^2.9'

Enable modules

The following have been separated into more commands for readability. These may be combined into fewer commands in practice.

drush pm-enable ctools
drush pm-enable purge
drush pm-enable cloudflare
drush pm-enable cloudflarepurger
drush pm-enable purge_ui
drush pm-enable purge_drush
drush pm-enable purge_queuer_coretags
drush pm-enable http_cache_control

If you enable the processors prior to configuring the cloudflare and purge modules, you will likely see this diagnostic error:

[error] diagnostics: ERROR: Purgers: There is no purger loaded which means that you need a module enabled to provide a purger plugin to clear your external cache or CDN.

This is ok. This issue will be addressed after configuring the cloudflare module and adding the cloudflarepurger to the purge module configuration. Or you may choose to configure those modules first, before enabling the processors. The following have been separated into more commands for readability. These may be combined into fewer commands in practice.

drush pm-enable purge_processor_cron
drush pm-enable purge_processor_lateruntime

Configure modules

You will need to configure the following modules:

  • cloudflare
  • purge
  • http_cache_control

cloudflare module

drush cset cloudflare.settings auth_using token

drush cset cloudflare.settings api_token XXXXXXXXXXXXXX

drush cset cloudflare.settings client_ip_restore_enabled true

Drupal UI Only: Presently, the Zone Selection can only be automatically configured via the Drupal UI. If/When a command line equivalent solution (via drush) is identified, these instructions will be added here to this documentation.

In the meantime, navigate to the Cloudflare Settings page (Configuration > Web services, or https://[SITE URL]/admin/config/services/cloudflare).

Select “Cloudflare” under the “Web services” category:

Select Authenticate using “Token”, and paste the API token into the “CloudFlare API Token” field.

Select the zone the API token is scoped for in the “Limit by zone name” drop-down under Current Zone Selection.

  • If the website is in a Self Service Cloudflare or Basic Cloudflare zone that you own, enter the name of that zone (example: some-site.some-unit.umich.edu would be under the Self Service zone name some-unit.umich.edu).
  • Otherwise, enter the Turnkey Cloudflare zone name.  For most people, this will be umich.edu, but it could also be umdearborn.edu, umflint.edu, some-unit.umich.edu, or some-domain.org.

Select “Next”.

Your API token should be scoped to a single Cloudflare zone, and you should see a message indicating that. Press “Finish”.

purge module

With the purge_drush module enabled, you can use purge-specific drush commands to configure the Purge module.

The list of available commands is available for reference in the README.md in the Purge module's git repository.

Add cloudflare purger:

drush ppadd cloudflare

http_cache_control module

Enable the http_cache_control module if it is not already:

drush pm-enable http_cache_control

Configure the module as follows:

  • Shared cache max age (s-maxage):  The recommended setting is "1 year" (31536000 seconds).

Check current configuration:

drush cget http_cache_control.settings cache.http.s_maxage

Change configuration to 1 year (31536000):

drush cset http_cache_control.settings cache.http.s_maxage 31536000

Confirm setting changed:

drush cget http_cache_control.settings cache.http.s_maxage

 

  • Browser cache maximum age (max-age):  By default, the cache maximum age is configured to ‘0’ (no caching). After setting s_maxage to 1 year, the recommended configuration for max-age when using external cache invalidation is in the range of 1 min - 5 min.  This is because the browser cache will mostly affect content editors so a short browser cache age is best to avoid confusion after content editing.

Check current configuration:

drush cget system.performance cache.page.max_age

Change configuration to 1 minute (60) or 5 minutes (300) as desired:

drush cset system.performance cache.page.max_age 300 --yes

Confirm new configuration:

drush cget system.performance cache.page.max_age

  • Cache-Control: must-revalidate

Confirm either in the UI that the box is checked for this, or confirm it is set to '1' via drush:

drush cget http_cache_control.settings cache.http.mustrevalidate

If necessary, set to 1:

drush cset http_cache_control.settings cache.http.mustrevalidate 1 --yes

Sites Hosted on U-M Pantheon

If your site is hosted on U-M Pantheon, there are additional module considerations to address: 

  1. You will also need to have the Pantheon Advanced Page Cache module installed, enabled, and configured in order to use this installation option. Please see the Pantheon documentation, Pantheon Advanced Page Cache: Drupal Cache Metadata on a Global CDN, and Setting and Clearing Custom Cache Tags in Drupal (v8.8+), for more information.
  2. The BigPipe module is incompatible with Pantheon and must be uninstalled.  Please see the Pantheon documentation, Drupal Modules with Known Issues, for more information.

Configure site User Agent

Add the following line to your site's settings.local.php or settings.php file.  This will configure Drupal to use a special User Agent that is allow-listed in Cloudflare for requests that Drupal makes (content Drupal fetches) of other university websites.  Replace YOUR-SITE-FQDN below with the fully qualified domain name for your site.

$settings['http_client_config']['headers']['User-Agent'] = 'Drupal (+https://drupal.org/); UMich; https://YOUR-SITE-FQDN/';

Web server cache headers for static assets

Sites Hosted on U-M Pantheon

The following is NOT applicable if your site is hosted on U-M Pantheon.  If your site is hosted elsewhere and is using Apache HTTPD Server, continue with the following.

If you are using Apache HTTP Server, add the following to Drupal's top-level .htaccess file.  

 

<IfModule mod_expires.c>
  ExpiresActive On
  ExpiresDefault A0

  <FilesMatch "\.(txt|xml|js)$">
    ExpiresDefault A31536000
  </FilesMatch>

  <FilesMatch "\.(css)$">
    ExpiresDefault A31536000
  </FilesMatch>

  <FilesMatch "\.(flv|ico|pdf|avi|mov|ppt|doc|mp3|wmv|wav|mp4|m4v|ogg|webm|aac|eot|ttf|otf|woff|woff2|svg)$">
    ExpiresDefault A31536000
  </FilesMatch>

  <FilesMatch "\.(jpg|jpeg|png|gif|swf|webp)$">
    ExpiresDefault A31536000
  </FilesMatch>
</IfModule>

<IfModule mod_headers.c>
  <FilesMatch "\.(txt|xml|js)$">
   Header set Cache-Control "max-age=31536000"
  </FilesMatch>

  <FilesMatch "\.(css)$">
   Header set Cache-Control "max-age=31536000"
  </FilesMatch>

  <FilesMatch "\.(flv|ico|pdf|avi|mov|ppt|doc|mp3|wmv|wav|mp4|m4v|ogg|webm|aac|eot|ttf|otf|woff|woff2|svg)$">
   Header set Cache-Control "max-age=31536000"
  </FilesMatch>

  <FilesMatch "\.(jpg|jpeg|png|gif|swf|webp)$">
   Header set Cache-Control "max-age=31536000"
  </FilesMatch>
</IfModule>
